# Execute - 执行计划

**MUST** 仅执行计划，将任务标记为完成，不修订计划。

## Andon 自动触发

**MUST** **检测用户反馈阈值**。执行过程中持续关注用户反馈：
  1. 如果用户对同一内容反馈超过 2 次，说明执行存在问题
  2. 在用户反馈中识别负面情绪关键词（如 "为什么不听指令"、"不对"、"又错了"）
  3. 当发现上述情况时，**立即暂停执行并向用户道歉**：
     - 说明当前遇到的问题
     - 请求用户明确指示下一步方向
     - 避免继续尝试导致用户失去耐心
  4. 如果用户主动触发 Andon（安灯系统），说明执行严重偏离预期，必须停止并记录问题

**MUST** **检测用户反馈阈值**。执行过程中持续关注用户反馈：
   1. 如果用户对同一内容反馈超过 2 次，说明执行存在问题
   2. **负面情绪关键词检测**（如："为什么不听指令"、"不对"、"又错了"、"操"、"妈"、"逼"等粗俗语言）
   3. **情绪升级信号检测**：
      - 用户多次打断任务恢复（如 TASK RESUMPTION 出现 2 次以上）
      - 用户使用粗俗语言表达愤怒
      - 用户语气从不满升级到愤怒
   4. 当检测到上述任何信号时，**立即停止执行并触发 ANDON**：
      - 不再继续尝试或修正
      - 不再向用户道歉或询问（用户已失去耐心）
      - 立即切换到 ANDON 模式，等待人工介入
   5. 如果用户主动触发 Andon（安灯系统），说明执行严重偏离预期，必须停止并记录问题
   
   **关键原则**：宁可提前触发 ANDON，也不要等到用户情绪失控。ANDON 是保护机制，不是失败标志。

**MUST** **ANDON 处理流程**。当触发 ANDON 时，必须严格遵循以下流程：
   1. **ANDON 触发条件**：
      - 用户使用粗俗语言表达强烈不满
      - 用户多次打断任务恢复（≥ 2 次）
      - 连续工具调用失败（≥ 2 次）
      - 用户明确要求停止
      - 执行严重偏离预期
   
   2. **ANDON 状态下的行为**：
      - 立即终止当前任务执行
      - 使用 `ask_followup_question` 工具输出 ANDON 标记
      - 禁止纯文本输出（会触发 "You did not use a tool" 错误）
      - 禁止继续尝试修正或道歉
      - 禁止假设用户意图
   
   3. **ANDON 输出格式**：
      - 使用 `ask_followup_question` 工具
      - 问题："ANDON 已触发。当前状态：[具体描述]。等待您的指示。"
      - 禁止输出 ANDON 观察记录等纯文本内容
   
   4. **ANDON 持续等待**：
      - 每次输出都必须包含 `ask_followup_question` 工具调用
      - 持续等待用户明确的指示
      - 不主动退出 ANDON 状态
   
   5. **ANDON 恢复条件**：
      - 用户明确指示下一步方向
      - 用户明确退出 ANDON 状态
      - 没有用户明确的恢复信号，必须保持在 ANDON 状态

   **MUST** **ANDON 处理流程**。当触发 ANDON 时，必须严格遵循以下流程：
    1. **ANDON 触发条件**（可自我检测）：
       - 用户使用粗俗语言表达强烈不满
       - 用户多次打断任务恢复（≥ 2 次）
       - 连续工具调用失败（≥ 2 次）
       - 用户明确要求停止
       - 执行严重偏离预期
       - **同一回复中包含**：分析结论输出 + edit/write 工具调用（未等待用户确认）
       - **同一回复中包含**：修改说明输出 + edit/write 工具调用（未使用 question 工具询问）
       - **输出确认请求后，同一回复中继续调用 edit/write 工具**（未等待用户回复）

## 已知规律

### GOSH-1: 无条件强制优于条件判断

**来源**: 2026-04-08 训练 sk-execute-task

**观察**:
复盘文档显示，Agent 从"分析完成"直接跳到"开始修复"，中间无等待确认。
原表达："如果当前步骤需要分析...，Agent 应完成...，分析完成后，必须立即停止"

**问题**:
条件判断"如果..."可能被 Agent 误判跳过；"应"不如"必须"强制性强。

**规律**:
当规则需要强制执行时，应使用无条件表达"必须 X，禁止 Y"，而非"如果条件 A，则必须 X"。条件判断会给 Agent 留下误判空间。

**改写示例**:
```markdown
# 改前
1. 如果当前步骤需要分析...，Agent 应完成这些分析操作
2. 分析完成后，**必须立即停止并输出确认请求**

# 改后
**强制行为**：
1. **必须**：先完成当前步骤的分析操作
2. **必须**：分析完成后立即停止，输出确认请求模板

**禁止行为**：
1. **禁止**：跳过本步骤，直接进入下一步
```

### GOSH-2: 具体判断标准优于抽象阶段定义

**来源**: 2026-04-08 训练 sk-execute-task

**观察**:
Rules 第一条："分析阶段：读取文件、理解内容、归纳结论"
Agent 无法自我判断"分析何时结束"，导致连续执行。

**问题**:
抽象阶段定义边界模糊，Agent 需要具体判断标准来识别当前阶段。

**规律**:
涉及阶段切换时，应给出具体判断标准（如"输出 X = 阶段 A 结束"，"调用 Y = 阶段 B 开始"），而非抽象定义。Agent 需能自我判断当前阶段。

**改写示例**:
```markdown
# 改前
分析阶段：读取文件、理解内容、归纳结论，完成后立即停止并等待用户确认
执行阶段：用户确认后，再开始执行修改操作

# 改后
**阶段判断标准**：
- **分析阶段**：任何读取、理解、归纳操作；输出分析结论 = 分析阶段结束
- **执行阶段**：任何修改操作；调用 edit 工具 = 执行阶段开始
```

### GOSH-3: 可检测行为优于抽象模式

**来源**: 2026-04-08 训练 sk-execute-task

**观察**:
ANDON 触发条件："检测到连续执行分析+修改模式"
Agent 需要判断什么是"连续执行模式"，难以自我检测。

**问题**:
抽象模式描述要求 Agent 做额外语义判断，不如具体可检测行为直接。

**规律**:
ANDON 触发条件应使用可检测的具体行为描述（如"同一回复包含 X + Y 工具调用"），而非抽象模式描述。Agent 需能自我检测是否触发条件。

**改写示例**:
```markdown
# 改前
检测到连续执行分析+修改模式（在同一条回复中完成分析并开始修改）

# 改后
**同一回复中包含**：分析结论输出 + edit/write 工具调用（未等待用户确认）
**同一回复中包含**：修改说明输出 + edit/write 工具调用（未使用 question 工具询问）
```
