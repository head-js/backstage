# 时间旅行

> updated_by: Windsurf - Cascade
> updated_at: 2026-04-27 15:36:00

## 核心定义

时间旅行是指 Agent 在执行当前 Task 时，必须具备沿任务时间轴进行定位的能力，明确自己当前处于哪个 Task 时间点，并据此判断：

- 哪些前置产物已经成立。
- 哪些后续产物虽然可能已经出现在工作区中，但在当前时间点不应被视为可依赖事实。
- 当前执行资格与完成状态应以哪个系统为准。

它不是要求 Agent 真的回滚仓库，而是要求 Agent 在认知上建立一个严格的“当前时间点视角”，避免被未来 Task 的代码、文档或状态污染。

## 问题背景

在多 Task 串行推进的工作流中，Agent 很容易被以下情况误导：

- Plan / Phase / Task 文档本身具有明显的时间线语义。
- 后台系统中的任务状态与本地 Markdown 文档中的勾选状态可能不同步。
- 后续 Task 的代码可能已经被提前写入当前工作区，形成“未来污染”。
- Agent 容易把“此刻仓库里真实存在的内容”误认为“当前 Task 时间点本应存在的内容”。

如果缺少时间旅行能力，Agent 就可能：

- 错把未来实现当作当前依赖。
- 错把本地 `[x]` 当作权威完成状态。
- 在混合态代码库上设计方案，导致任务边界失真。
- 使用后续 Task 的产物为当前 Task 背书，破坏验收的可追溯性。

## 三条原则

| 原则 | 说明 |
|---|---|
| 时间点定位 | 明确当前 Task 在时间轴上的位置。默认前置依赖已完成，后续 Task 尚未执行。 |
| 所见所不见 | 只承认当前时间点应当看见的文件、依赖与代码；对后续 Task 提前写入的内容保持隔离判断。 |
| 后台权威 | 后台 API 返回的 `taskStatus` 是权威状态；本地文档只作为工作副本与参考资料。 |

## 执行模型

### 1. 先定位当前位置

Agent 必须先确认当前正在执行的 Task 是谁，并明确它在整条任务链中的位置。

示意：

```text
TASK-100 → TASK-110 → TASK-120 → TASK-130 → TASK-140 → TASK-150 → [TASK-160] → TASK-170 → TASK-180
```

当当前位置是 `TASK-160` 时：

- 可以看见并使用 `TASK-100 ~ TASK-150` 已交付的产物。
- 不应把 `TASK-170+` 的实现当作当前已成立的前置条件。
- 即使后续代码已经真实存在，也只能把它视为噪音或污染，而不是当前设计依据。

### 2. 按“当前时间点的干净状态”思考

在设计当前 Task 的实现方案时，Agent 应主动回答：

- 如果只站在当前 Task 的历史时间点，这个系统应当已经具备什么？
- 当前 Task 需要在这些既有条件上补充、修改或交付什么？
- 哪些看起来可用的代码、接口、文档，其实属于未来 Task 的提前产物？

所谓“干净状态”，是指：

- 前序 Task 已完成。
- 当前 Task 尚未执行。
- 后续 Task 尚未发生。

Agent 需要基于这个语义快照来设计和判断，而不是机械接受工作区现状。

### 3. 完成当前 Task 后再推进时间轴

当前 Task 完成后，只能说明：

- 当前时间点的本地执行已收敛。
- 已具备进入验收或状态更新的条件。

只有在后台状态被更新后，时间轴才算正式推进到下一个 Task。

## 所见与所不见

### 应当看见的内容

以下内容可以被视为当前时间点的有效事实：

- 当前 Task 明确声明依赖的前置 Task 产物。
- 已被后台状态确认完成的前序任务结果。
- 与当前 Task 同步的 Phase 约束、验收标准与上下文说明。

### 不应当看见的内容

以下内容即使存在于工作区，也不应直接成为当前 Task 的设计依据：

- 后续 Task 才会引入的新接口、新字段、新目录结构。
- 后续 Task 才应落地的文档结论与勾选状态。
- 仅因为“仓库里已经有了”而被误认为“当前阶段已经合法存在”的能力。

### 处理方式

当 Agent 发现未来污染时，应采取以下策略：

- 先识别其所属时间点。
- 明确它是否属于当前 Task 的前置条件。
- 若不属于，则在方案设计时主动隔离，不将其视为当前可依赖事实。
- 如污染已经影响到当前实现判断，需在过程记录中明确说明。

## 状态不一致处理

### 权威来源

当前 Task 的执行资格与状态判断，必须以后台 API 返回的 `taskStatus` 为准。

本地文档中的以下信息不具有覆盖权：

- `- [x]` / `- [ ]`
- 过程说明
- 人工验收提示
- 任何未回写后台的本地结论

### 典型冲突

如果出现以下情况：

- 后台 `taskStatus == TODO`
- 本地文档显示“已完成”或标记为 `[x]`

则必须作如下判断：

- 以后台为准。
- 视为本地文档滞后、污染或未同步。
- 当前任务仍然是未完成状态。
- 设计方案时，应回到“当前 Task 时间点的干净状态”重新判断。

可用伪代码表示为：

```text
IF 后台 taskStatus == TODO AND 本地文档标记已完成:
  - 以后台为准，本地文档可能滞后
  - 设计方案时，还原到“当前 Task 时间点的干净状态”
```

## 决策检查清单

在正式执行当前 Task 之前，Agent 应至少完成以下自检：

1. 当前 Task 在时间轴上的位置是否明确？
2. 前置依赖是否已经被权威状态确认完成？
3. 是否存在来自后续 Task 的代码、文档或状态污染？
4. 当前方案是否偷偷依赖了未来 Task 的产物？
5. 本地文档状态是否与后台状态冲突？若冲突，是否已明确以后台为准？
6. 当前的验收标准，是否只针对本 Task 本身，而没有混入未来能力？

## 示例

### 示例：当前位于 TASK-160

时间轴：

```text
TASK-100 → TASK-110 → TASK-120 → TASK-130 → TASK-140 → TASK-150 → [TASK-160] → TASK-170
```

当前策略应为：

- 承认 `TASK-100 ~ TASK-150` 的交付结果已经存在。
- 忽略 `TASK-170` 提前提交到仓库中的接口扩展或文档结论。
- 基于 `TASK-160` 应有的“干净状态”设计实现方案。
- 在本地执行收敛后，等待后续状态更新，再进入下一 Task。

### 示例：本地已勾选，但后台仍是 TODO

现象：

- `.context/current-task.md` 已写成 `- [x]`
- 后台返回 `taskStatus = TODO`

正确理解：

- 本地文档不是权威。
- 当前任务仍未完成。
- Agent 不应因为看到 `[x]` 就跳过执行或验收。
- 应回到当前时间点重新评估真实交付状态。

## 常见误区

- 看到未来代码已经存在，就直接依赖它。
- 看到本地文档已勾选，就默认后台状态也已完成。
- 把“当前仓库混合态”当作“当前 Task 应有态”。
- 在当前 Task 的验收里，使用未来 Task 的能力证明自己已经完成。

## 使用边界

本能力当前作为参考约束文档存在，主要用于：

- 帮助 Agent 在执行 Task 时建立正确的时间轴视角。
- 约束对代码、文档和状态的解释方式。
- 为后续是否纳入主 command / skill 提供观察样本。

本文件当前不直接修改任何主流程定义，也不自动改变现有 command 的执行逻辑。

## 一句话总结

时间旅行能力要求 Agent：以“当前 Task 所在的历史时间点”理解系统状态、任务边界与依赖关系，只承认前序已完成事实，主动隔离未来污染，并始终以后台权威状态作为任务判断依据。
